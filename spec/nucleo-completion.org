#+title: nucleo-completion 仕様

* 目的

~nucleo-completion~ は Emacs の completion style として動作し、Rust dynamic
module と ~nucleo-matcher~ による fuzzy matching / scoring を提供する。

ビルド済み dynamic module が存在しない環境でも、Emacs Lisp の fallback による
絞り込みは維持する。ただし fallback では Rust-backed scoring による並び替えは
提供しない。

* 公開インターフェース

** Completion Style

~completion-styles~ には ~nucleo~ を指定する。

#+begin_src emacs-lisp
  (require 'nucleo-completion)
  (add-to-list 'completion-styles 'nucleo)
#+end_src

Corfu など通常バッファ補完でも ~nucleo~ の scoring を使う場合は、global
~completion-styles~ に ~nucleo~ を追加する。Migemo/pyim などの regexp expander を
通常バッファ補完へ適用したくない場合は、対象 mode hook で
~nucleo-completion-regexp-functions~ を buffer-local に nil にする。

** package-vc

ビルド環境がないユーザー向けに、~main~ branch に
~bin/<target-triple>/~ 配下の dynamic module を含める。

#+begin_src emacs-lisp
  (package-vc-install
   '(nucleo-completion
     :url "https://github.com/kn66/nucleo-completion.el"))
#+end_src

~package-vc~ で ~main~ を取得すれば、そのまま dynamic module をロードできる状態にする。

** Custom Variables

- ~nucleo-completion-max-highlighted-completions~
   - lazy highlight が使えない UI 向けに、先頭何件を eager highlight するかを指定する。
   - 負数や整数以外が設定された場合、実行時は 0 として扱う。
- ~nucleo-completion-regexp-functions~
   - 各検索 term を追加 regexp に展開する関数リスト。
   - Migemo や pyim などの言語固有 matcher は本体へ直接組み込まず、ユーザー設定でこの hook に登録する。
- ~nucleo-completion-persistent-regexp-cache-size~
   - 既定値は nil で、regexp expander の結果は completion pass 内だけ cache する。
   - 正の整数が設定された場合、term、関数リスト、~completion-ignore-case~、
     ~default-directory~ を key にして completion pass をまたいで cache する。
   - expander が key に含まれない可変状態へ依存する場合は使わない。
- ~nucleo-completion-long-candidate-threshold~
   - Rust module で score する候補の最大文字数を指定する。
   - 既定値は 4096。nil の場合は候補長に関係なくすべて score する。
   - しきい値を超える候補は score/sort せず、filter だけを行い、score 済み候補の後ろへ入力順で追加する。
- ~nucleo-completion-long-candidate-regexp-threshold~
   - regexp expander を適用する候補の最大文字数を指定する。
   - 既定値は 4096。nil の場合は候補長に関係なく regexp expander を適用する。
   - しきい値を超える候補は ~nucleo-completion-regexp-functions~ 由来の regexp を filter と highlight の両方で使わず、組み込み fuzzy subsequence matcher だけを使う。
- ~nucleo-completion-long-candidate-highlight-threshold~
   - match highlight を適用する候補の最大文字数を指定する。
   - 既定値は 4096。nil の場合は候補長に関係なく match highlight を適用する。
   - しきい値を超える候補は候補文字列を走査する match highlight を省略する。
- ~nucleo-completion-sort-ties-by-length~
   - Nucleo score が同じ候補を、短い候補が先になるように並び替えるかを指定する。
- ~nucleo-completion-sort-ties-alphabetically~
   - Nucleo score が同じ候補をアルファベット順に並び替えるかを指定する。
   - ~nucleo-completion-sort-ties-by-length~ も有効な場合は、長さで比較した後にアルファベット順を使う。
- ~nucleo-completion-highlight-score-bands~
   - 候補全体の face を score band ごとに変えるかを指定する。
   - 既定値は nil で、既存の match 部分 highlight のみを行う。
- ~nucleo-completion-high-score-ratio~
   - 最高 score に対して high-score 候補とみなす最小比率を指定する。
   - 完全一致の単語を含む候補は、この比率に関係なく high-score として扱う。
   - 実行時は 0.0 から 1.0 の範囲に丸める。数値以外が設定された場合は既定値
     0.85 として扱う。
- ~nucleo-completion-high-score-emphasis~
   - high-score 候補へ追加する強調 face を指定する。
   - 値は ~bold~、~underline~、その両方、または nil のリストとする。
   - 既定値は ~(bold underline)~。

** Dynamic Module Functions

Rust dynamic module がロードされた場合、次の関数が利用可能になる。

- ~nucleo-completion-candidates~
   - pattern、candidate list、ignore-case、同点 sorting options、highlight limit を受け取る。
   - match する候補を Nucleo score 順に並べ、各 entry を ~(candidate score indices)~ として返す。
   - ~indices~ は ~highlight limit~ 件までの上位候補だけに付与し、Nucleo が選んだ一致位置の文字 index リストとする。

* Matching Semantics

** Term Split

入力 pattern は空白文字で分割する。空 term は無視する。

例:

#+begin_src text
  "foo bar" -> ("foo" "bar")
#+end_src

** Term 間の関係

term 同士は AND 条件である。すべての term が候補に一致する必要がある。

** Term 内の関係

各 term では、標準 fuzzy subsequence matcher と
~nucleo-completion-regexp-functions~ が返す regexp matcher を OR 条件で扱う。

例:

#+begin_src text
  term = "nihon"
  fuzzy: n.*i.*h.*o.*n
  regexp expander: 日本

  candidate は fuzzy または 日本 regexp のどちらかに一致すればよい。
#+end_src

* Processing Flow

#+begin_src mermaid
  flowchart TD
      A[completion input] --> B[completion boundaries]
      B --> C[extract prefix and needle]
      C --> D{needle empty?}
      D -->|yes| E[return candidates without nucleo filtering]
      D -->|no| F{Rust module loaded?}
      F -->|yes| G{regexp expanders produced extra regexps?}
      G -->|yes| H[Rust module scores fuzzy-matchable candidates in parallel]
      H --> I[Elisp regexp filter for candidates not matched by module]
      I --> J[promote regexp-only matches before Nucleo-scored matches]
      G -->|no| K[Rust module filters and sorts with parallel scoring]
      F -->|no| L[Elisp fallback filter]
      K --> M[highlight top candidates]
      J --> M
      L --> M
      E --> M
      M --> N[completion result]
#+end_src

* Module Loading

Dynamic module は次の順で探索する。

1. ~locate-library~ で見つかる ~nucleo-completion-module~
2. ~locate-library~ で見つかる platform library name
3. ~bin/<target-triple>/~
4. package directory root
5. ~target/release/~
6. ~target/debug/~

Platform library name:

- Linux: ~libnucleo_completion_module.so~
- macOS: ~libnucleo_completion_module.dylib~
- Windows: ~nucleo_completion_module.dll~

Prebuilt module は ~bin/<target-triple>/~ に配置する。

読み込み可能な module file が存在しても ~module-load~ が失敗した場合、その失敗は
~nucleo-completion-module-load-errors~ に ~(FILE . MESSAGE)~ として保存する。
~nucleo-completion-report-module-load-errors~ が non-nil の場合は ~message~ でも報告する。

* Performance

Rust dynamic module は Emacs から受け取った候補を一度 Rust 文字列へ変換し、
候補数が多い場合は ~nucleo-matcher~ scoring を worker thread に分配する。
Emacs の value は worker thread へ渡さず、score 計算後に main thread で
completion list を再構築する。worker thread は固定 size の batch を atomic counter
経由で取得するため、候補文字列の長さや scoring cost が偏っても worker 間の負荷が
偏りにくい。

~nucleo-matcher~ の ~Matcher~ は scratch memory を持つため、single-thread 経路では
thread-local に再利用し、parallel worker では worker ごとに作成する。

Emacs 側の completion entry point は ~while-no-input~ で filtering を包み、
入力が到着した場合は直近の結果を返せる場合だけ返す。ただし Rust module 呼び出しは
同期的に実行されるため、非常に大きな候補集合では module 呼び出しが戻るまで Emacs を
占有し得る。

~nucleo-completion-regexp-functions~ の結果は completion pass ごとに cache する。
これにより、regexp expansion の必要判定、候補 filtering、highlighting の各段階で
同じ term の expander を重複実行しない。

検索文字列の term 分割と fuzzy subsequence regexp の構築も completion pass ごとに
cache する。候補 filtering、score-band 判定、eager/lazy highlight が同じ needle と
term を繰り返し参照しても、同じ純粋な文字列処理を再実行しない。

score と precomputed indices の lookup table は highlight で必要になった場合だけ
作成する。highlight 件数が 0 で lazy highlight も無効な場合は、module 結果を
filter/sort 済み候補リストとして使うだけにし、追加の hash table 作成を避ける。
lazy highlight では候補文字列の property-free key を一度だけ作り、score と indices
の lookup に再利用する。

score-band highlight が有効な場合、high-score 判定で使う単語完全一致 regexp も
completion pass ごとに cache する。同じ needle の上位候補を複数 highlight しても、
term ごとの regexp 文字列を候補ごとに組み立て直さない。

Rust module が使える場合、regexp expander が有効でも先に Rust module で fuzzy
filtering/scoring を行う。module が fuzzy match として返した候補には Emacs Lisp
側の regexp filtering を再適用せず、module から漏れた候補だけを
~nucleo-completion-regexp-functions~ 由来の matcher を含む regexp 条件で確認する。
これにより、Migemo/pyim など大きい regexp を生成する expander で候補全体を毎回
走査する負荷を減らす。

~nucleo-completion-persistent-regexp-cache-size~ が正の整数の場合、
~nucleo-completion-regexp-functions~ の結果を completion pass をまたいで cache する。
cache entry 数が上限に達した場合は cache 全体を clear してから新しい結果を保存する。
辞書や expander の設定を変更したユーザーは
~nucleo-completion-clear-persistent-regexp-cache~ で明示的に cache を破棄できる。

~nucleo-completion-sort-ties-by-length~ または
~nucleo-completion-sort-ties-alphabetically~ が有効な場合、tie sorting は Rust module
内で行う。score 降順の後に、設定に応じて length 昇順、alphabetical 昇順、最後に
元の index 昇順で安定した順序にする。Rust module が利用できない場合の
Emacs Lisp fallback では score も sort も行わない。

~nucleo-completion-long-candidate-threshold~ が non-nil の場合、しきい値を
超える候補は Rust module に渡さず、Emacs Lisp 側の fuzzy/regexp filter だけで
一致判定する。長い候補は Nucleo score、tie sorting、score-band highlight の対象外
とし、通常の score 済み候補の後ろに入力順で追加する。これにより、非常に長い行が
少数含まれる completion table で module scoring が候補全文を走査して停止時間を
伸ばすことを避ける。

~nucleo-completion-long-candidate-regexp-threshold~ が non-nil の場合、しきい値を
超える候補には ~nucleo-completion-regexp-functions~ 由来の regexp を適用しない。
これにより、Migemo/pyim など大きな regexp を非常に長い行へ適用して Emacs Lisp 側
の filter/highlight が停止時間を伸ばすことを避ける。長い候補でも組み込み fuzzy
subsequence matcher には引き続きかける。

~nucleo-completion-long-candidate-highlight-threshold~ が non-nil の場合、
しきい値を超える候補には match highlight を適用しない。これにより、非常に長い
候補に対して eager highlight や lazy highlight が候補全文を走査することを避ける。

* Highlight

候補表示では、Rust module が利用可能な場合は ~nucleo-matcher~ が選んだ一致位置を
~completions-common-part~ face で強調する。module が利用できない fallback では、
各 term の greedy subsequence match を強調する。

~nucleo-completion-regexp-functions~ が返す regexp match も
~completions-common-part~ face で強調する。regexp が空文字列に一致する場合は、
無限ループしないように次の位置へ進める。

~nucleo-completion-highlight-score-bands~ が non-nil の場合は、match 部分
highlight に加えて候補全体へ score band の face を付ける。全 term が
単語として完全一致する候補、または ~nucleo-completion-high-score-ratio~ に基づき
最高 score に近い候補は ~nucleo-completion-high-score-face~ を使う。それ以外の
scored candidates は ~nucleo-completion-low-score-face~ を使う。
regexp expander でしか一致しない候補は辞書一致として高優先度に扱い、
~nucleo-completion-high-score-face~ を使う。
high-score 候補には ~nucleo-completion-high-score-emphasis~ で指定された追加 face
も付与する。既定では ~bold~ と ~underline~ の両方を付与するが、テーマが
太字や下線を別用途に使う場合は片方、または両方を無効化できる。

lazy highlight をサポートする UI では ~completion-lazy-hilit-fn~ を設定する。
それ以外では ~nucleo-completion-max-highlighted-completions~ 件だけ eager highlight
する。

* Edge Cases

- 空入力では独自 scoring/filtering を行わず、既存 completion metadata の sort 関数を尊重する。
- 無効 regexp、nil、非文字列を返す regexp function は無視する。
- persistent regexp cache は既定で無効にし、有効時も上限を持つ。
- persistent regexp cache は nil 結果も cache し、同じ key で失敗を繰り返さない。
- regexp expander でしか一致しない候補は、Rust module sorting 後も落とさない。
- regexp expander でしか一致しない候補は、辞書一致として Nucleo-scored 候補の前に昇格する。
- regexp expander でしか一致しない候補には、score band highlight で high-score face を使う。
- 同点 sorting は Nucleo score が同じ候補にだけ適用し、異なる score の順序は維持する。
- Rust module がない場合も、Elisp fallback により絞り込みは行う。ただし fallback は
  入力順を保ち、score、sort、score band highlight は行わない。
- case sensitivity は ~completion-ignore-case~ に従う。
- Corfu など通常バッファ補完では ~nucleo~ scoring を使い、regexp expander だけを使わない構成をサポートする。
- highlight 件数や score ratio に範囲外の値が設定されても、filtering 中に
  signal せず安全な値へ正規化する。
