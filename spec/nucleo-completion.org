#+title: nucleo-completion 仕様

* 目的

~nucleo-completion~ は Emacs の completion style として動作し、Rust dynamic
module と ~nucleo-matcher~ による fuzzy matching / scoring を提供する。

ビルド済み dynamic module が存在しない環境でも、Emacs Lisp の fallback による
絞り込みと設定済み tie sorting は維持する。ただし fallback では Rust-backed
scoring は提供しない。

* 公開インターフェース

** Completion Style

~completion-styles~ には ~nucleo~ を指定する。

#+begin_src emacs-lisp
  (require 'nucleo-completion)
  (add-to-list 'completion-styles 'nucleo)
#+end_src

Corfu など通常バッファ補完でも ~nucleo~ の scoring を使う場合は、global
~completion-styles~ に ~nucleo~ を追加する。Migemo/pyim などの regexp expander を
通常バッファ補完へ適用したくない場合は、対象 mode hook で
~nucleo-completion-regexp-functions~ を buffer-local に nil にする。

~completion-try-completion~ は、単一 term の通常 fuzzy 入力では Emacs 組み込み
~completion-flex-try-completion~ の merge 挙動を使う。空白で分割される複数 term や
~nucleo-completion-regexp-functions~ が追加 regexp を返す入力では、
~nucleo-completion-all-completions~ と同じ候補集合を使って結果を作る。候補はあるが
共通 prefix を伸ばせない場合は、nil ではなく元の入力と point を返し、
~try-completion~ 側でも「候補あり」と扱えるようにする。point より後ろでも同じ
completion boundary に含まれる text は、Emacs 組み込み completion style と同様に
filtering、highlight、~try-completion~ の検索条件へ含める。結果を返す直前には
completion table 自身の ~try-completion~ も確認し、
~completion-table-with-terminator~ などの組み込み table wrapper が付与する終端文字や
point 後方の suffix merge を維持する。filtered ~try-completion~ は入口で計算した
completion boundary 情報を再利用し、同じ table wrapper に対する boundary 問い合わせを
繰り返さない。ただし completion text が既に変わらない場合は
point だけを進める結果へ refine せず、組み込み style と同じ exact match semantics を保つ。
Emacs 組み込みの ~completion-flex-nospace~ が non-nil の場合、completion field に
space、tab、newline などの空白類を含む入力は組み込み ~flex~ と同じく
~nucleo~ style では扱わず、後続の completion style へ委ねる。

** Installation

MELPA package には prebuilt dynamic module を含めない。MELPA recipe は
~nucleo-completion.el~、~src/~、~Cargo.toml~、~Cargo.lock~ を含め、必要なユーザーが
インストール後にローカル build できるようにする。

#+begin_src emacs-lisp
  (package-vc-install
   '(nucleo-completion
     :url "https://github.com/kn66/nucleo-completion.el"))
#+end_src

~package-vc~ で ~main~ を取得した場合も、dynamic module がない環境では Elisp fallback
で動作する。

Prebuilt module を使いたいユーザー向けに、明示コマンド
~nucleo-completion-install-module~ を提供する。このコマンドは現在の platform 向け
GitHub Release asset だけを取得し、SHA256 を検証してから
~nucleo-completion-module-directory/<release-tag>/<target-triple>/~ 配下へ保存する。
既定の release tag は ~nucleo-completion-required-module-version~ に対応する
~v<version>~ とする。package install 時や既定の load 時には network access を
行わない。すでに Rust module が current Emacs process で loaded の場合も installed
file は更新するが、loaded module は差し替えず、次回 Emacs 起動後に新しい module を
使う。

設定ファイルから明示的に自動 install したいユーザー向けに、
~nucleo-completion-ensure-module~ を提供する。この関数は module API が未ロードの場合は
既存 search path から module load を試みる。それでも module API が使えない場合、または
older installed release の module だけが loaded された場合は
~nucleo-completion-install-module~ を呼び、現在の platform 向け package-matched
prebuilt module を install する。Lisp から呼ばれた場合は、設定に書いたことを明示
opt-in とみなして確認なしで install する。~M-x~ から呼ばれた場合は download や
置き換え前に確認する。

** Custom Variables

- ~nucleo-completion-max-highlighted-completions~
   - lazy highlight が使えない UI 向けに、先頭何件を eager highlight するかを指定する。
   - 負数や整数以外が設定された場合、実行時は 0 として扱う。
- ~nucleo-completion-regexp-functions~
   - 各検索 term を追加 regexp に展開する関数リスト。
   - 各関数は term 文字列を 1 つ受け取り、空でない regexp 文字列、
     空でない regexp 文字列の list、または nil を返す。
   - 空文字 regexp は term を全候補一致にしてしまうため無視する。
   - Migemo や pyim などの言語固有 matcher は本体へ直接組み込まず、ユーザー設定でこの hook に登録する。
- ~nucleo-completion-report-regexp-function-errors~
   - 既定値は nil。~nucleo-completion-regexp-functions~ の関数が error を signal した場合、
     その expander を silent に無視する。
   - non-nil の場合、error を signal した expander ごとに session 中一度だけ
     ~display-warning~ で報告する。
- ~nucleo-completion-regexp-minimum-term-length~
   - regexp expander を適用する最小 term 長。
   - 既定値は 2。Migemo などで 1 文字ローマ字入力が非常に広い regexp へ展開され、
     ~consult-line~ のような大きな候補集合を重くすることを避ける。
   - 0 または 1 に設定すると、短い term でも従来どおり expander を実行する。負数や整数以外は
     実行時に既定値 2 として扱う。
- ~nucleo-completion-regexp-only-match-priority~
   - Rust module 経路で regexp expander だけに一致した候補を、Nucleo-scored fuzzy
     候補の前後どちらに置くかを指定する。
   - 既定値は ~non-ascii~。非 ASCII 文字を含む regexp-only 候補は辞書一致として前に置き、
     ASCII-only の regexp-only 候補は後ろに置く。これにより、Migemo/pyim などの
     cross-script match を高く保ちながら、ローマ字 regexp が拾った ASCII コマンド名などを
     直接 fuzzy match より上に出しにくくする。
   - ~before~ はすべての regexp-only 候補を前に置く。~after~ はすべての regexp-only
     候補を後ろに置く。
- ~nucleo-completion-sort-ties-by-history~
   - 同じ score の候補を、現在の completion history で最近使われた候補が先になるように並び替えるかを指定する。
   - Rust module が利用できない fallback では、絞り込まれた全候補を同じ score として扱う。
   - minibuffer では現在の ~minibuffer-history-variable~ を使う。
   - 通常バッファでは ~corfu-history-mode~ が有効で ~corfu-history~ が利用可能な場合に ~corfu-history~ を使う。
   - minibuffer history entry は completion boundary の prefix を除いた候補文字列として比較する。
   - ~corfu-history~ entry は候補文字列そのものとして比較し、候補側の text property は無視する。
- ~nucleo-completion-sort-ties-by-length~
   - 同じ score の候補を、短い候補が先になるように並び替えるかを指定する。
   - Rust module が利用できない fallback では、絞り込まれた全候補を同じ score として扱う。
- ~nucleo-completion-sort-ties-alphabetically~
   - 同じ score の候補をアルファベット順に並び替えるかを指定する。
   - Rust module が利用できない fallback では、絞り込まれた全候補を同じ score として扱う。
   - 履歴 sorting、~nucleo-completion-sort-ties-by-length~ も有効な場合は、履歴、長さ、アルファベット順の順に比較する。
- ~nucleo-completion-highlight-score-bands~
   - 候補全体の face を score band ごとに変えるかを指定する。
   - 既定値は nil で、既存の match 部分 highlight のみを行う。
- ~nucleo-completion-high-score-ratio~
   - 最高 score に対して high-score 候補とみなす最小比率を指定する。
   - 完全一致の単語を含む候補は、この比率に関係なく high-score として扱う。
   - 実行時は 0.0 から 1.0 の範囲に丸める。数値以外が設定された場合は既定値
     0.85 として扱う。
- ~nucleo-completion-high-score-emphasis~
   - high-score 候補へ追加する強調 face を指定する。
   - 値は ~bold~、~underline~、その両方、または nil のリストとする。
   - 既定値は ~(bold underline)~。
- ~nucleo-completion-module-directory~
   - ~nucleo-completion-install-module~ が prebuilt module を保存する user directory。
- ~nucleo-completion-module-release-repository~
   - prebuilt module を取得する GitHub repository。形式は ~OWNER/REPO~。
- ~nucleo-completion-module-release-tag~
   - prebuilt module を取得する GitHub Release tag。
   - 既定値は nil で、~nucleo-completion-required-module-version~ に対応する
     ~v<version>~ release asset を使う。
   - ~latest~ を設定した場合は、GitHub の ~latest/download~ URL を使う。
   - 文字列を設定した場合は、その tag の release asset を使う。
- ~nucleo-completion-module-install-policy~
   - 既定値は ~manual~ で、自動 prompt は出さない。
   - ~prompt~ の場合、interactive session で module がないときだけ一度確認 prompt を出す。
   - nil の場合は prompt を出さない。

** Dynamic Module Functions

Rust dynamic module がロードされた場合、次の関数が利用可能になる。

- ~nucleo-completion-candidates~
   - pattern、candidate list、ignore-case、同点 sorting options、highlight limit を受け取る。
   - ~(CANDIDATES TOP-INFO FULL-SCORES)~ の bundle を返す。
   - ~CANDIDATES~ は match する候補を Nucleo score 順に並べた list とする。
   - ~TOP-INFO~ は上位候補の metadata list で、各 entry を
     ~(candidate score indices)~ とする。
   - ~indices~ は ~highlight limit~ 件までの上位候補だけに付与し、Nucleo が選んだ一致位置の文字 index リストとする。
   - ~FULL-SCORES~ は full score が要求された場合だけ返す、~CANDIDATES~ と平行な
     score list とする。
- ~nucleo-completion-candidates-with-history~
   - ~nucleo-completion-candidates~ と同じ結果を返すが、candidate list と平行な history rank list も受け取る。
   - history rank は Nucleo score が同じ候補の tie-breaker としてだけ使う。
- ~nucleo-completion-module-version~
   - Rust dynamic module の package version を文字列で返す。
   - Elisp 側の ~nucleo-completion-required-module-version~ と一致しない module は
     scoring backend として使わず、session ごとに一度 warning を出す。

* Matching Semantics

** Term Split

入力 pattern は空白文字で分割する。空 term は無視する。

例:

#+begin_src text
  "foo bar" -> ("foo" "bar")
#+end_src

** Term 間の関係

term 同士は AND 条件である。すべての term が候補に一致する必要がある。

** Term 内の関係

各 term では、標準 fuzzy subsequence matcher と、term 長が
~nucleo-completion-regexp-minimum-term-length~ 以上の場合に限って
~nucleo-completion-regexp-functions~ が返す regexp matcher を OR 条件で扱う。

例:

#+begin_src text
  term = "nihon"
  fuzzy: n.*i.*h.*o.*n
  regexp expander: 日本

  candidate は fuzzy または 日本 regexp のどちらかに一致すればよい。
#+end_src

* Processing Flow

#+begin_src mermaid
  flowchart TD
      A[completion input] --> B[completion boundaries]
      B --> C[extract prefix and needle]
      C --> D{needle empty?}
      D -->|yes| E[return candidates without nucleo filtering]
      D -->|no| F{Rust module loaded?}
      F -->|yes| G{regexp expanders produced extra regexps?}
      G -->|yes| H[Rust module scores fuzzy-matchable candidates]
      H --> I[Elisp regexp-only filter for candidates not matched by module]
      I --> J[merge regexp-only matches by configured priority]
      G -->|no| K[Rust module filters and sorts matches]
      F -->|no| L[Elisp fallback filters and tie-sorts]
      K --> M[highlight top candidates]
      J --> M
      L --> M
      E --> M
      M --> N[completion result]
#+end_src

* Module Loading

Dynamic module は次の順で探索する。

1. ~nucleo-completion-module-directory/<configured-release>/<target-triple>/~
2. ~bin/<target-triple>/~
3. package directory root
4. ~target/release/~
5. ~target/debug/~
6. ~locate-library~ で見つかる ~nucleo-completion-module~
7. ~locate-library~ で見つかる platform library name
8. ~nucleo-completion-module-directory/<older-release>/<target-triple>/~

Versioned install、同梱 bin、local build を generic ~load-path~ module より優先する。
これは古い global module が package-matched module や local build を shadow し、一度
load されると同一 Emacs session で差し替えられない状況を避けるためである。
Older installed release は fallback としてのみ使う。読み込まれた場合は session ごとに
一度 warning を出し、package-matched module の install と Emacs restart を促す。

Platform library name:

- Linux: ~libnucleo_completion_module.so~
- macOS: ~libnucleo_completion_module.dylib~
- Windows: ~nucleo_completion_module.dll~

Prebuilt module は GitHub Releases に
~nucleo-completion-module-<target-triple>.<extension>~ として配置する。
~nucleo-completion-install-module~ は同名の ~.sha256~ asset も取得して検証する。
~prebuilt-modules~ workflow は、入力 release tag が Cargo package version と
~nucleo-completion-required-module-version~ に対応する ~v<version>~ と一致することを
検証し、CI と同じ Rust/Elisp verification gates を通してから build/publish する。
手動で module を配置する場合は、~bin/<target-triple>/~ または上記の user module
directory を使える。

読み込み可能な module file が存在しても ~module-load~ が失敗した場合、その失敗は
~nucleo-completion-module-load-errors~ に ~(FILE . MESSAGE)~ として保存する。
~nucleo-completion-report-module-load-errors~ が non-nil の場合は ~message~ でも報告する。

Module load に成功しても、module が ~nucleo-completion-module-version~ を提供しない、
または返す version が ~nucleo-completion-required-module-version~ と一致しない場合は、
その module を compatible API とみなさない。loaded module は Emacs process 内で
差し替えられないため、warning で matching module の install と Emacs restart を促し、
completion は Elisp fallback を使う。

* Performance

Rust dynamic module は Emacs から受け取った候補を Rust 文字列へ変換しながら
~nucleo-matcher~ で score を計算する。match しなかった候補の文字列は保持せず、
match した候補だけ保持する。tie-break が不要な経路では Emacs value、score、元 index
だけを持つ compact な構造を使い、history / length / alphabetical のいずれかが必要な
経路だけ tie-break 用 key（history rank、length、alphabetical key）も保持する。
これにより、全候補分の Emacs value / Rust string vector を作らず、alphabetical sorting
が有効な場合も match した候補だけ key を作る。

~nucleo-matcher~ の ~Matcher~ は scratch memory を持つため、module call 間で
thread-local に再利用する。highlight 用の match indices は score 計算後の上位
~highlight-limit~ 件だけで再計算し、その時だけ Emacs value から Rust string を
再取得する。上位候補の間では UTF-32 変換 buffer と indices buffer を再利用する。

Emacs 側の completion entry point は ~while-no-input~ で filtering を包み、
入力が到着した場合は直近の結果を返せる場合だけ返す。ただし Rust module 呼び出しは
同期的に実行されるため、非常に大きな候補集合では module 呼び出しが戻るまで Emacs を
占有し得る。

Rust module は module 呼び出しの開始時と scoring 後に ~input-pending-p~ を確認する。
入力が pending の場合は、候補なしと区別できる sentinel bundle を返す。Elisp 側はその
sentinel を受け取ると filtering を中断し、~while-no-input~ の中断時と同じく直近の
filtered result を再利用する。non-Unicode scrub が有効な module call でも、sentinel
bundle は候補復元処理を通さず、そのまま interrupt として扱う。

~nucleo-completion-regexp-functions~ の結果は completion pass ごとに cache する。
これにより、regexp expansion の必要判定、候補 filtering、highlighting の各段階で
同じ term の expander を重複実行しない。
~nucleo-completion-regexp-functions~ が nil の場合、regexp expansion の必要判定は
検索文字列を term 分割せずに nil を返す。fuzzy-only の regexp group 構築と
precomputed indices による highlight も regexp expander lookup を省略する。
~completion-regexp-list~ に追加する fuzzy-only regexp は、term ごとの regexp group を
組み立てず flat list として直接生成する。

Rust module 経路で regexp-only match を救済する場合、expander が regexp を返した
term では module がすでに処理した fuzzy subsequence regexp を再実行せず、追加 regexp
だけを評価する。expander を返さない term は、regexp-only match と組み合わせるために
標準 fuzzy subsequence regexp を評価する。Rust module の結果件数が元候補数以上の場合は、
module から漏れた候補がないため regexp-only 用の regexp 構築と候補 scan を省略する。
regexp-only の追加候補がない場合は、score 計算や before/after 分割を行わず module
bundle をそのまま返す。
regexp-only の追加候補があっても、~full-scores~ 拡張と synthetic top-info のどちらも
不要な経路では最大 score を参照しない。
Rust module が空結果を返した場合は、module 候補との重複排除が不要なため seen table を
作らず regexp-only 候補を直接 filter する。
module 候補がある場合も seen table の初期 size 計算のためだけの module 候補全走査は
行わず、table へ登録する pass だけで重複排除を準備する。
score band highlight が無効な場合は、~full-scores~ を保持する経路でも regexp-only
追加候補への score property 付与 copy を作らない。
score band highlight が有効で ~full-scores~ も保持する経路では、regexp-only 追加候補の
score property 付与と ~full-scores~ 用 score list 作成を同じ pass で行う。
lazy score-band など ~full-scores~ がある経路では regexp-only 追加候補の score を
score property から取得できるため、regexp-only match 用の synthetic top-info を構築しない。
~nucleo-completion-regexp-only-match-priority~ が ~before~ または ~after~ の場合は、
全 regexp-only 候補の位置が一括で決まるため、候補ごとの ASCII 判定と before/after 分割
pass を省略する。
eager highlight 件数が 0 の場合、regexp-only match 用の synthetic top-info は利用されないため
構築しない。score band highlight が無効な場合も、regexp-only match 用の synthetic
top-info は score 分類に使われないため構築しない。

検索文字列の term 分割と fuzzy subsequence regexp の構築も completion pass ごとに
cache する。候補 filtering、score-band 判定、eager/lazy highlight が同じ needle と
term を繰り返し参照しても、同じ純粋な文字列処理を再実行しない。
Emacs Lisp fallback の regexp filtering は一致候補だけを直接 collect し、不要な
中間 pair list を作らない。
filtered ~try-completion~ の共通 prefix 計算では、text property を除いた候補文字列を
hash table で重複排除し、候補数が増えた場合の二乗的な duplicate scan を避ける。
text property を持たない候補はそのまま key として使い、property がある候補だけ
copy して stripping する。候補が 1 件または 2 件だけの場合は直接処理し、重複排除用の
hash table を構築しない。3 件以上の候補で hash table を使う場合も、初期 size 計算の
ためだけの候補 list 全走査は行わない。

score と precomputed indices の lookup table は highlight で必要になった場合だけ
作成する。highlight 件数が 0 で lazy highlight も無効な場合は、module 結果を
filter/sort 済み候補リストとして使うだけにし、追加の hash table 作成を避ける。lookup
table 作成時も、初期 size 計算のためだけの ~top-info~ 全走査は行わない。
Rust module が top-info の precomputed highlight indices を計算する場合は、
highlight 対象候補の間で UTF-32 変換 buffer と indices buffer を再利用し、候補ごとの
小さな vector allocation を避ける。
plain な list completion table を prefix なしで扱う場合、その table は
~all-completions~ の base-size tail marker を持たないため、base-size 検出用の末尾走査を
省略して候補数に比例する余分な pass を避ける。この fast path では
~all-completions~ に渡す Nucleo-generated regexp list も使われないため、fallback filter
が直接適用する regexp group を事前に組み立てない。
lazy highlight の score-band 判定では、Elisp wrapper が候補復元後に付与した
~nucleo-completion-score~ text property を使う。これにより、長い共通 prefix を持つ
候補リストで高コストになり得る ~equal~ hash table 構築を避けつつ、non-Unicode scrub
後に同じ text へ潰れる候補も元 object へ復元してから score を付けられる。古い module や
test double がすでに score property を返す場合も同じ lazy highlight 経路を使う。
score property 補完は candidate list と full score list を一度だけ平行走査し、既に
正しい score property を持つ候補は元 object のまま保持する。
内部の score 付き module filter は候補 list を基準に alist を作り、malformed bundle で
full score が不足していても score が欠けた候補を落とさない。

~nucleo-completion-scrub-non-unicode-candidates~ が nil の場合、Rust module に
候補を渡す前の non-Unicode codepoint 走査は行わない。通常の completion table は
Unicode 文字列だけを含むため、この既定値により Emacs Lisp 側で候補全文をもう一度
走査するコストを避ける。module が ~unicode-string-p~ error で候補を拒否した場合は、
その module call だけ一度 non-Unicode scrub を有効にして retry する。scrub は候補全文の
走査を伴い、大きい候補集合では数十 ms 以上になり得るため、retry 成功後に全 completion
へ自動的に sticky 適用しない。Consult などの frontend が Unicode 上限を超える disambiguation 文字を候補へ
付与することが分かっている環境では、この option を non-nil にすることで初回から
scrubbed copy を module に渡し、戻り値では元の candidate object を復元する。scrubbed
candidate restore map は置換が実際に必要になった場合だけ作り、初期 size 計算のための
候補 list 全走査は行わない。

score-band highlight が有効で、かつ eager/lazy highlight のどちらかを実行する場合、
high-score 判定で使う単語完全一致 regexp も completion pass ごとに cache する。
同じ needle の上位候補を複数 highlight しても、term ごとの regexp 文字列を候補ごとに
組み立て直さない。lazy highlight では UI から後で呼ばれる highlight lambda が
この cache を保持する。highlight を行わない経路では、この cache 用 hash table も作らない。
eager/lazy highlight のどちらでも、score ratio だけで high-score と判定できる候補だけを
処理している間は exact-word cache を作らず、単語完全一致への fallback が必要になった
最初の候補で遅延作成する。
score ratio だけで high-score と判定できる候補では、単語完全一致 regexp の評価を省略する。
score または max score が数値でない場合は score ratio 判定を行わず、単語完全一致の
high-score 判定へ fall back する。

2026-05-22 に Linux / GNU Emacs 30.2 の ~emacs-completion-bench~ で確認した。
同 benchmark の candidate set は 95,653 items / 約 3.59MB / 平均 37.58 bytes。
local checkout を読むため、次の command を使った。

#+begin_src sh
  EMACS_COMPLETION_BENCH_NUCLEO_DIR=/home/nobu43/.emacs.d/lisp/nucleo-completion.el \
    timeout 240s ./runbench
#+end_src

match した候補だけを保持し、alphabetical key も match 後にだけ作る module 経路では
次の結果だった。

| ~(nucleo . nucleo-completion-highlight-score-bands)~ | 0.607707254 |   0 | 0.0     |

同じ session で、全 match の Rust string を保持する経路は ~nucleo . default~ が
約 0.719s、全候補の lower-case key を作る indexed alphabetical 経路は

production module は match した候補だけで scoring/sorting を完結させる。

non-Unicode scrub の sticky retry は採用しない。次の batch command で、clean な
50,000 candidates / 120 chars を 20 回処理した。

#+begin_src sh
  emacs -Q --batch -L . -L target/release --eval \
    '(progn
       (require (quote benchmark))
       (require (quote nucleo-completion))
       (let* ((base (make-string 120 ?x))
              (candidates (make-list 50000 base))
              (nucleo-completion-scrub-non-unicode-candidates nil)
              (nucleo-completion--force-scrub-non-unicode-candidates nil))

                        (benchmark-run 20

         (princ (format "scrub-enabled=%S\n"
                        (benchmark-run 20

#+end_src

結果は ~scrub-disabled=(0.047267 0 0.0)~、
~scrub-enabled=(2.777861 4 0.027629)~ だった。scrub は clean candidates でも
full scan を伴うため、module が ~unicode-string-p~ error を返した call だけ retry
し、以後の completion に自動で sticky 適用しない。

Rust module が使える場合、regexp expander が有効でも先に Rust module で fuzzy
filtering/scoring を行う。module が fuzzy match として返した候補には Emacs Lisp
側の regexp filtering を再適用せず、module から漏れた候補だけを
~nucleo-completion-regexp-functions~ 由来の matcher を含む regexp 条件で確認する。
これにより、Migemo/pyim など大きい regexp を生成する expander で候補全体を毎回
走査する負荷を減らす。

~nucleo-completion-sort-ties-by-history~、
~nucleo-completion-sort-ties-by-length~、または
~nucleo-completion-sort-ties-alphabetically~ が有効な場合、Rust module 経路では
tie sorting を Rust module 内で行う。score 降順の後に、設定に応じて
history rank 昇順、length 昇順、alphabetical 昇順、最後に元の index 昇順で
安定した順序にする。history rank は
minibuffer では現在の ~minibuffer-history-variable~ の値から Emacs Lisp 側で構築し、
completion boundary の prefix を除いた candidate 文字列と比較する。通常バッファでは
~corfu-history-mode~ が有効で ~corfu-history~ が利用可能な場合に ~corfu-history~
から構築し、candidate 文字列と比較する。candidate 側の text property は history
比較では無視し、candidate key をコピーせず ~equal~ hash lookup で照合する。候補が
0 件または 1 件の場合は history tie-break が発生しないため、history rank list/table
を構築しない。history entries が空、または prefix に
一致する文字列 entry がない場合も一致し得る rank がないため、空の rank table を
構築しない。rank table 作成時も、初期 size 計算のためだけの history entries 全走査は
行わない。
古い Rust module が history-aware API を持たない場合は、full score を要求して
Emacs Lisp 側で同じ score の候補だけ history rank で並び替える。full score に
隣接同点がない場合は、history で順序が変わらないため sort 用の中間 list 作成と
sort を省略する。Elisp 側 comparator は 1 比較につき candidate 2 件の rank を一度だけ
lookup し、その結果を後続の tie-break 判定に再利用する。highlight 用 ~top-info~ も
隣接同点がない場合は同じく sort 用の中間 list 作成と sort を省略する。古い module
から返る full score が候補数より短い
malformed bundle でも、score が欠けた候補は落とさず、欠けた score は ~nil~ として
保持する。score が欠けた候補が複数隣接する場合は ~nil~ 同点として扱い、history
tie sort の対象にする。
Rust module に渡る history rank list が候補数より短い場合、欠けた rank は未登録候補と
同じ扱いにし、module 側で panic しない。
Rust module が利用できない場合の Emacs Lisp fallback では score は計算しないが、
絞り込まれた全候補を同じ score として扱い、history rank 昇順、length 昇順、
alphabetical 昇順、最後に元の index 昇順で並び替える。fallback 結果が 0 件または
1 件の場合は、順序が変わらないため history table 構築や sort key 作成を省略する。

候補長に基づく事前分割は行わず、全候補を Rust module に渡す。通常の completion
table ではこの経路が最速で、Emacs Lisp 側で候補ごとに ~length~ を取る pre-scan と、
長い候補だけを fallback regexp filter に回す分岐を避けられる。

* Highlight

候補表示では、Rust module が利用可能な場合は ~nucleo-matcher~ が選んだ一致位置を
~completions-common-part~ face で強調する。module が利用できない fallback では、
各 term の greedy subsequence match を強調する。

~nucleo-completion-regexp-functions~ が返す regexp match も
~completions-common-part~ face で強調する。regexp が空文字列に一致する場合は、
無限ループしないように次の位置へ進める。

~nucleo-completion-highlight-score-bands~ が non-nil の場合は、match 部分
highlight に加えて候補全体へ score band の face を付ける。全 term が
単語として完全一致する候補、または ~nucleo-completion-high-score-ratio~ に基づき
最高 score に近い候補は ~nucleo-completion-high-score-face~ を使う。それ以外の
scored candidates は ~nucleo-completion-low-score-face~ を使う。
regexp expander でしか一致しない候補は ~nucleo-completion-high-score-face~ を使う。
high-score 候補には ~nucleo-completion-high-score-emphasis~ で指定された追加 face
も付与する。既定では ~bold~ と ~underline~ の両方を付与するが、テーマが
太字や下線を別用途に使う場合は片方、または両方を無効化できる。

lazy highlight をサポートする UI では ~completion-lazy-hilit-fn~ を設定する。
それ以外では ~nucleo-completion-max-highlighted-completions~ 件だけ eager highlight
する。
lazy highlight は UI から渡された候補文字列を直接変更せず、highlight 済み copy を返す。
lazy highlight と score band を併用する場合、eager highlight 件数が 0 で
~top-info~ が空でも、~full-scores~ の先頭を最高 score として high/low band を判定する。

* Edge Cases

- 空入力では独自 scoring/filtering/highlight を行わず、既存 completion metadata の
  sort 関数を尊重し、前回の filtering/lazy highlight state をクリアする。この経路では
  completion pass 用の regexp/term cache も作らない。
- ~completion-flex-nospace~ が non-nil の場合、completion field 内に空白類を含む入力は
  ~nucleo~ が候補を返さない。completion boundary の外側の prefix に含まれる空白は
  判定対象にしない。この即終了経路でも completion pass 用の regexp/term cache は
  作らない。
- ~nucleo~ が候補を返さず後続 completion style に処理を譲る場合は、
  ~nucleo-completion--filtering-p~ と ~completion-lazy-hilit-fn~ をリセットする。
- filtering 中かどうかを示す metadata 調整 state は buffer-local とし、別 buffer の
  completion metadata に sort 抑制を漏らさない。
- completion table が候補リストの末尾 cdr に base-size marker を返す場合は、
  filtering 前に proper list として扱い、返却時、~try-completion~ 後の reusable state、
  interrupt reuse 時に marker を保持する。
- interrupt reuse 用の前回結果は display highlight を付ける前の候補リストとして保持し、
  入力途中で中断された再表示に古い match face を混ぜない。
- ~try-completion~ が組み込み flex の成功結果だけを返し、候補一覧を作らない場合は、
  metadata 調整用の filtering state だけを有効にし、前回の reusable result と
  lazy highlight state は破棄する。
- 空 regexp、無効 regexp、nil、非文字列を返す regexp function は無視する。
- error を signal した regexp function は無視する。debug 用に
  ~nucleo-completion-report-regexp-function-errors~ が non-nil の場合は、関数ごとに一度だけ warning を出す。
- Rust module version が package の required version と一致しない場合は module scoring を使わず、fallback へ戻る。
- regexp expander でしか一致しない候補は、Rust module sorting 後も落とさない。
- 既定では、regexp expander でしか一致しない非 ASCII 候補は Nucleo-scored 候補の前に置き、
  ASCII-only 候補は後ろに置く。
- regexp expander でしか一致しない候補の前後関係は
  ~nucleo-completion-regexp-only-match-priority~ で変更できる。
- regexp expander でしか一致しない候補には、score band highlight で high-score face を使う。
- precomputed highlight index が範囲外または整数でない場合は無視し、UI 表示を落とさない。
- 同点 sorting は Nucleo score が同じ候補にだけ適用し、異なる score の順序は維持する。
- history sorting は minibuffer history も Corfu history も利用できない補完では no-op とする。
- Rust module がない場合も、Elisp fallback により絞り込みは行う。fallback は score を
  計算せず、sort option が無効な場合は入力順を保つ。sort option が有効な場合は、
  全候補を同 score として history、length、alphabetical の順に並び替える。
  score band highlight は行わない。
- case sensitivity は ~completion-ignore-case~ に従う。
- non-Unicode codepoint を scrub して module に渡した候補も、full score が要求された場合は
  復元後の候補に ~nucleo-completion-score~ text property を保持する。
- scrub 後に clean candidate と scrubbed candidate が同じ text へ潰れる場合も、
  module が score 付き copy を返す経路で元候補の対応関係を保つ。
- scrub 有効時でも Rust module が interrupt sentinel を返した場合は、候補復元用の
  queue/hash を作らずに前回結果 reuse へ進む。
- point 後ろの同一 completion boundary 内 text は filtering と ~try-completion~ で
  無視しない。
- completion text が既に exact match の場合、~try-completion~ は point だけを進める
  cons を返さない。
- Corfu など通常バッファ補完では ~nucleo~ scoring を使い、regexp expander だけを使わない構成をサポートする。
- highlight 件数や score ratio に範囲外の値が設定されても、filtering 中に
  signal せず安全な値へ正規化する。
